# Template Rendering

`hkey template render` lets you generate config files or `.env` files by substituting secret values directly into a template. No secrets need to be stored in your repository — just the template.

---

## How it works

```
template.env                          .env (rendered output)
─────────────────────────────         ─────────────────────────────
DATABASE_URL={{ /prod/myapp:db/url }} DATABASE_URL=postgres://user:s3cr3t@db:5432/myapp
API_KEY={{ /prod/myapp:api-key }}     API_KEY=sk-live-abc123
JWT_SECRET={{ /prod/myapp:jwt-secret }}  JWT_SECRET=supersecret
```

The template is a plain text file with `{{ /namespace:secret-name }}` placeholders. The CLI fetches each secret from Hierarkey, base64-decodes the value, and writes the substituted file.

---

## Template syntax

Placeholders use double braces containing a full secret reference:

```
{{ /namespace:secret-name }}
```

Whitespace inside the braces is ignored — these are all equivalent:

```
{{ /prod/myapp:db-password }}
{{/prod/myapp:db-password}}
{{  /prod/myapp:db-password  }}
```

The same placeholder can appear multiple times in a template — it will be replaced everywhere.

---

## Usage

```
hkey template render --file <template-file> [--output <output-file>]
```

| Flag | Required | Description |
|------|----------|-------------|
| `--file` | Yes | Path to the template file |
| `--output` | No | Output file path; defaults to stdout |

## Examples

### Render a .env file

Template (`template.env`):

```bash
# Application environment — generated by hkey template render
DATABASE_URL={{ /prod/myapp:database-url }}
REDIS_PASSWORD={{ /prod/myapp:redis-password }}
API_KEY={{ /prod/myapp:api-key }}
JWT_SECRET={{ /prod/myapp:jwt-secret }}

# Non-secret config (no placeholder needed)
APP_ENV=production
LOG_LEVEL=info
```

Render it:

```bash
export HKEY_SERVER_URL=https://hierarkey.internal
export HKEY_TOKEN=hkat_...

hkey template render --file template.env --output .env
```

The output `.env` file contains the real secret values. Never commit it.

### Render to stdout and pipe

```bash
hkey template render --file template.env | docker run --env-file /dev/stdin myapp:latest
```

### Render a config file

Template (`config.toml.tpl`):

```toml
[database]
url = "{{ /prod/myapp:database-url }}"

[redis]
password = "{{ /prod/myapp:redis-password }}"

[auth]
jwt_secret = "{{ /prod/myapp:jwt-secret }}"
```

```bash
hkey template render --file config.toml.tpl --output /etc/myapp/config.toml
```

### Render an nginx config

Template (`nginx.conf.tpl`):

```nginx
server {
    listen 443 ssl;
    ssl_certificate     /etc/ssl/certs/myapp.crt;
    ssl_certificate_key /etc/ssl/private/myapp.key;

    location / {
        proxy_pass http://backend;
        proxy_set_header X-Internal-Token {{ /prod/myapp:internal-service-token }};
    }
}
```

```bash
hkey template render --file nginx.conf.tpl --output /etc/nginx/sites-enabled/myapp
nginx -s reload
```

## Using templates in CI/CD

In a GitHub Actions workflow:

```yaml
- name: Authenticate with Hierarkey
  run: |
    echo "${{ secrets.HIERARKEY_CI_PRIVATE_KEY }}" > /tmp/ci.pem
    chmod 600 /tmp/ci.pem
    TOKEN=$(hkey auth sa token --method keysig --name ci-myapp --private-key /tmp/ci.pem --print access-token)
    rm /tmp/ci.pem
    echo "HKEY_TOKEN=${TOKEN}" >> "$GITHUB_ENV"
  env:
    HKEY_SERVER_URL: ${{ secrets.HIERARKEY_SERVER }}

- name: Render config
  env:
    HKEY_SERVER_URL: ${{ secrets.HIERARKEY_SERVER }}
  run: |
    hkey template render --file config.toml.tpl --output config.toml
```

## Using templates in Docker

Add template rendering to your container startup:

```dockerfile
FROM alpine:3.19

RUN apk add --no-cache curl

RUN curl -sSL https://github.com/hierarkey/hierarkey/releases/latest/download/hkey-linux-amd64 \
    -o /usr/local/bin/hkey && chmod +x /usr/local/bin/hkey

COPY config.toml.tpl /etc/myapp/config.toml.tpl
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

```bash
#!/bin/sh
# entrypoint.sh
set -e

# Render the config file from the template
hkey template render \
  --file /etc/myapp/config.toml.tpl \
  --output /etc/myapp/config.toml

# Unset credentials before exec
unset HKEY_TOKEN

exec "$@"
```

---

## Constraints

- Secret values must be valid UTF-8 text. Binary secrets (certificates, keys) cannot be embedded in text templates directly — use the `certificate_key_pair` type and write the value to a file separately.
- Hierarkey fetches each unique secret reference once. Duplicate placeholders are substituted without additional API calls.
- The rendered output is written atomically (the output file is not partially written on error).

## Template files in git

Check in template files — they contain no secrets:

```gitignore
# .gitignore

# Rendered output files — never commit
.env
config.toml
/etc/myapp/*.conf

# Template files — safe to commit
# template.env        (DO commit)
# config.toml.tpl     (DO commit)
```
